# 🚀 Pushkin NetOps Engine

**Pushkin** — это высокопроизводительный асинхронный демон для сетевой автоматизации, разработанный для провайдеров (ISP), дата-центров и крупных корпоративных сетей. Система отходит от медленных последовательных моделей настройки и использует принцип **«Fire-and-Forget»** (выстрелил и забыл) с концепцией **Eventual Consistency** (конечной согласованности).

## 🎯 Философия: «Стреляй первым, проверяй потом»
Классические инструменты (Ansible, Netmiko) ждут подтверждения от устройства (`prompt#`) после каждой введенной строки. В сетях с большими задержками или при массовых работах это становится критическим «бутылочным горлышком».

**Pushkin** меняет правила игры:
1. **Залп (Execution):** Моментально отправляет полный пакет команд в SSH-канал через неблокирующий `select`.
2. **Верификация:** Не тратит время SSH-сессии на ожидание подтверждения. Вместо этого запускается фоновый цикл примирения (Reconciliation Loop) через SNMP или Telemetry.
3. **Откат (Rollback):** Если цепочка настроек прервалась, система использует pre-fire бэкапы для автоматического восстановления состояния сети.

## 🛠 Ключевые возможности

- **Высокая скорость:** Адаптивная пачечная отправка команд через SSH.
- **Абстракция вендоров:** Поддержка Multi-vendor через **Jinja2 шаблоны**, хранящиеся в Redis.
- **Оркестрация VLAN:** Автоматический поиск свободного ID, атомарное бронирование и «протяжка» сервиса по цепочке устройств (Access -> Core).
- **Надежность и безопасность:** 
    - **Dry Run**: Предварительная проверка синтаксиса и доступности узлов.
    - **Распределенные блокировки**: Redis-locks предотвращают одновременную работу нескольких воркеров с одним IP.
    - **Умные ретраи**: Иерархия ошибок (проблемы сети vs ошибки авторизации).
- **Мониторинг:** Дашборд на Tailwind CSS в реальном времени и уведомления в Telegram.

## 🏗 Архитектура

- **API (FastAPI)**: Принимает намерения (Intents) от внешних систем.
- **Broker (Redis)**: Очереди задач, паспорта устройств и блокировки в реальном времени.
- **Worker (Python/Paramiko)**: Независимые процессы, выполняющие SSH-сессии.
- **nstat (в разработке)**: Интеграция с NetBox для получения актуальной топологии.

## 🚀 Быстрый старт

### Требования
- Docker & Docker Compose
- Telegram Bot Token (опционально для уведомлений)

### Запуск
1. **Клонируйте репозиторий:**
   ```bash
   git clone https://github.com
   cd pushkin
   ```

2. **Настройте окружение:**
   Создайте файл `.env` на основе шаблона:
   ```text
   TG_TOKEN=123456:ABC-DEF
   TG_CHAT_ID=987654321
   ```

3. **Запустите систему:**
   ```bash
   docker-compose up -d --build
   ```

4. **Интерфейс:**
   Откройте `http://localhost:8000` для доступа к дашборду Pushkin.

## 🛡 FAQ для сетевых инженеров

**В: Что если SSH-сессия оборвется на середине отправки команд?**  
О: Пушкин фиксирует аудит-лог. Если сессия прервана, задача помечается как `failed`. Так как перед «стрельбой» делается **Pre-fire Backup**, вы можете восстановить исходное состояние в один клик.

**В: Не «положит» ли это CPU старых коммутаторов?**  
О: Нет. Мы используем настраиваемые `chunk_size` и `read_timeout`. Вы можете задать профили для «медленного» железа, чтобы не забивать их буфер ввода.

**В: Это безопасно для продакшена?**  
О: Да. Благодаря атомарным бронированиям (SET NX) в Redis, Пушкин гарантирует, что два воркера никогда не займут одну и ту же VTY-линию или один и тот же VLAN ID.
